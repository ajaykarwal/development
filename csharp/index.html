Code conventions serve the following purpose:

  • A consistent look to the code
  • Code that is easier to read and follow
  • Facilitates the maintenance of code

Naming

  • Choose easily readable names, use good English
  • Favour readability over brevity
  • Do not use underscores, hyphens, or other non alphanumeric characters
  • Do not use Hungarian notation
  • Avoid using identifiers that conflict with language keywords
  • Avoid using abbreviations or contractions
  • Avoid using acronyms that are not widely understood and accepted
{% highlight c# %} // Good HorizontalAlignment CanScrollHorizontally Surname // Bad AlignmentHorizontal ScrollableX StrSurname {% endhighlight %}

Capitalization

  • Only capitalize the first letter of each word or acronym
  • A special exception is made for two-letter acronyms in which both letters are capitalized

Use upper camel case for (UpperCamelCase):

  • Types and namespaces
  • Methods, properties and events
  • Constants
  • Enum members
  • Fields, when not private

Use lower camel case for (lowerCamelCase):

  • Local variables
  • Parameters

Use underscore lower camel case for (_lowerCamelCase):

  • Fields, when private

Use upper camel case prefixed with I for interfaces (IUpperCamelCase).

Use upper camel case prefixed with T for generic types (TUpperCamelCase).

{% highlight c# %} // Capitalization conventions HtmlTag ResolveUrl IOStream htmlTag resolveUrl ioStream // UpperCamelCase namespace CreativeJar.Standards public class Class public string Field; public string Property { get; set; } public void Method() public enum Enum { ItemOne, ItemTwo, ItemThree } // lowerCamelCase public void Method(int parameter) { var localVariable = parameter + 1; } // _lowerCamelCase private string _field; // IUpperCamelCase public interface INaming // TUpperCamelCase public class Class where TName : class {% endhighlight %}

Formatting

  • Format code for legibility
  • Use the default editor settings (smart indent, four space indents, tabs saved as spaces)
  • Write only one statement per line
  • Write only one declaration per line
  • Avoid long lines, continue statements on the next line
  • Indent continuation lines
  • Use blank lines to separate methods and properties
  • Use blank lines to improve legibility
  • Braces should be on their own line unless opening and closing on the same line
{% highlight c# %} using System.IO; using System.Linq; namespace CreativeJar.Standards { public class Formatting { public string Property1 { get; set; } public string Property2 { get { return null; } set { Method(); Set(value); } } public int Method() { var variable = new[] { 1, 2, 3 }; for (var i = 0; i < variable.Length; i++) { variable[i] *= variable[i]; } return variable.Any() ? variable.Last() : default(int); } private void Set(string value) { using (var stream = File.Open(value, FileMode.Open)) using (var streamReader = new StreamReader(stream)) { Property1 = streamReader.ReadToEnd(); } } } } {% endhighlight %}

Layout

  • Classes should implement a uniform code structure
  • One type per file should be used, with certain exclusions
    • Enums can be collected into a single file but the name and location should be clear
    • Generic and non-generic implementations of the same type can exist in the same file
  • Regions should be used sparingly only to group related logic only, such as interface implementation
{% highlight bash %} ├─ Namespace | └─ Class | ├── Enums | ├── Constants | ├── Static fields | ├── Fields | ├── Constructor | ├── Properties & indexers | ├── Methods | ├── Interface implementations | └── Nested types {% endhighlight %}
